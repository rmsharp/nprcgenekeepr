## Structural Observations

1. **The tracker systematically lags the code, but only for *resolved* work — not the reverse.** Exactly 2 of 21 open issues are fully done (#14, #8); both were resolved during the Shiny-module conversion (Sessions 20–25) and never closed — the same failure that left #34 open. Recommendation already in flight (S61 gotcha): **read the code before treating any backlog item as implementation work.** This audit operationalizes that for the whole tracker.

2. **The auto-generated-ID cluster (#38, #32, #26, + dam-ID #31) is one feature split across four issues.** #32 and #26 are near-duplicates (both: replace hardcoded leading-`U` detection with a configurable hook). #38 adds the *format-generation* side; #31 is the dam-identification consumer of the same "U == unknown" convention. A single design ("introduce `isGeneratedUnknownId()` + `set/getAutoIdFormat()`, route all `^U` sites through it") would resolve all four. **Recommend consolidating** (keep one as the umbrella, reference the others) — matches S61 suggested-next #3.

3. **Old feature requests (#10, #11, #12, #13, #28) are genuinely open and large.** These need external systems (Oracle, ARMS), external data (molecular kinship, location transactions), or new simulation machinery (forward GVA). They are correctly open and are *not* staleness candidates — none should be closed.

4. **The adversarial verify pass changed the answer on a third of the close candidates.** Without it, #1 would have been wrongly closed (re-burying a gap the owner explicitly documented in 2020) and #8 would have been closed without its test-coverage caveat surfaced. For an outward-facing, owner-trust action like closing issues, the refute-the-close step is load-bearing, not ceremony.
